## 方向一：搜索加速

### 1.1 尝试：collision_checker numpy 向量化

**动机：** `collision_checker.py` 的碰撞检测是 Hybrid A\* 搜索中调用最频繁的函数，内层有两层 Python 循环（遍历4条边 × 遍历采样点）。直觉上用 numpy 向量化应该能提速。

**实现：** 把内层循环改为 `np.linspace` 生成采样点，再用批量索引 `grid_map[ix_arr, iy_arr]` 一次查表，替换原来的逐点 Python 循环。

**测试结果：比纯 Python 更慢。**

**原因分析：**
每次碰撞检测只沿 4 条车身边各采样约 10-15 个点，总共 40-60 个点。numpy 向量化在数据量大时才有优势，因为它的固定开销（数组创建、类型转换、内存分配）在小数据量时远大于收益。对于几十个点的查表操作，原生 Python 循环反而更快。

**结论：放弃 numpy 向量化，回退到纯 Python 版本。**

这个教训很典型：向量化不是银弹，要看数据规模。

---

### 1.2 尝试：RS 曲线直接替代 Hybrid A\* 搜索

**动机：** RS（Reeds-Shepp）曲线是支持前进+倒车的最短路径，能生成从起点到终点的完整几何路径。看到 `reeds_shepp.py` 里有 `rs_path()` 函数，自然想到：能不能直接用 RS 路径代替 Hybrid A\* 搜索？规划速度会快很多。

**尝试：** 调用 `rs_path(start_pos, goal_pos, turning_radius)` 直接生成路径点序列，跳过 Hybrid A\* 搜索。

**结果：失败——路径直接穿过障碍物。**

**根本原因：**
RS 曲线是**纯几何计算**，它只考虑起点和终点的位姿，以及车辆的最小转弯半径。它完全不知道地图上有障碍物，生成的路径是"自由空间中的最短路径"，而不是"避开障碍物的路径"。

在泊车场景中，车位两侧有停放的车辆，RS 曲线直接从起点到终点走最短路，会径直穿过邻车。

**结论：RS 曲线不能替代搜索，只能做启发式。**

---

### 1.3 尝试：RS 曲线作为启发式 h(n)

**动机：** 既然 RS 曲线本身是有转向约束的最短路径长度，用它作为 Hybrid A\* 的启发式 h(n) 比欧氏距离更准确——欧氏距离完全忽略转向约束，会低估实际代价。

**实现：** 费了相当大的功夫实现了完整的 RS 路径族：
- CSC 族：LSL、RSR、LSR、RSL（圆弧-直线-圆弧）
- CCC 族：RLR、LRL（圆弧-圆弧-圆弧）
- 每种族通过 time_flip（时间翻转）和 reflect（左右镜像）扩展为 24 种变体，共覆盖 48 种路径类型
- 最终取所有候选中的最短路径长度作为 h(n)

为了避免每步都算 RS 距离（计算量大），只在车辆距终点 `euclidean < 5m` 时才切换，否则仍用欧氏距离：

```python
h = rs_distance(next_state, self.goal, self.turning_radius) if euclidean < 5.0 else euclidean
```

**测试结果：**
- RS 启发式让搜索在最后几米的方向感更准
- 但整体搜索时间提升**不到 8%**，几乎可以忽略

**根本原因（事后分析）：**
搜索速度的真正瓶颈在状态空间大小，而不是启发式质量。Hybrid A\* 的 `dt=1.0`，每步走 1m，状态空间规模由 `(地图大小 / 分辨率)³` 决定。改善启发式只是让搜索"方向更准"，但要展开的节点数量本质上没有大幅减少。

**两项加速手段合计提升不到 8%，微不足道。**

**结论：RS 启发式保留在代码里（方向上正确），但实际效果有限。搜索加速这个方向没有取得实质性突破。**

### 1.4 尝试：双向 Hybrid A\* 搜索

**动机：** 双向搜索是加速图搜索的经典方法——从起点和终点同时展开搜索，两棵搜索树在中间相遇，理论上把搜索空间从 O(b^d) 降到 O(b^(d/2))，对大状态空间有显著加速效果。

**实现思路：** 维护两个 open/closed list，分别从起点（forward）和终点（backward）展开 Hybrid A\* 搜索。每次展开一个节点后，检查对方搜索树的 closed list 中是否有状态落在同一个离散格子里，若有则认为两棵树相遇，拼接路径。

**结果：失败——相遇点出现路径突变（折角）。**

**根本原因：**

Hybrid A\* 的状态是 $(x, y, \theta)$，方向 $\theta$ 是路径连通性的关键。两棵搜索树各自独立展开，forward 树的末端朝向和 backward 树的末端朝向几乎不可能完全对齐。当两棵树在某个格子里"相遇"时：
- forward 树末端的 $\theta_f$（车头朝某个方向）
- backward 树末端的 $\theta_b$（从终点反向搜索，车头朝另一个方向）

二者之间存在角度差，直接拼接会产生不可执行的折角——车辆需要在相遇点瞬间改变朝向，违反运动学约束。

这是双向 Hybrid A\* 的本质难点：在 SE(2) 空间里，两棵树的"相遇"不只需要位置重合，还需要朝向兼容，而朝向兼容的概率极低。解决方法需要在相遇点附近做专门的朝向对齐（如用 RS 曲线连接两端），实现复杂度大幅上升。

**结论：放弃双向搜索，代码未保留。** 在 SE(2) 状态空间里，双向搜索的实现难度远高于二维格子情形，工程代价超过收益。

## 方向二：路径平滑

Phase 3 的路径由离散运动原语拼接，在转向切换处有曲率突变（折角）。路径平滑的目标是消除这些折角，让路径更平滑、更便于控制器跟踪。

### 2.1 尝试：整体梯度下降平滑

**思路：** 参考 Coursera 自动驾驶课程的做法，对整条路径所有点同时施加两种力进行迭代：
- **平滑力**：牵引相邻三点趋向共线，消除折角
- **障碍物排斥力**：保持路径远离障碍物边界

**实现：** 用距离变换（`distance_transform_edt`）预计算距离场，每次迭代用数值差分估计梯度，更新所有路径点的 (x, y) 坐标。theta 保留原值不修改。

**结果：失败——theta 方向全乱，路径无法执行。**

**根本原因：**
Hybrid A\* 路径包含换挡点（前进→倒车）。在换挡点附近，theta 值是"前进方向"，而换挡后车辆开始倒车，theta 应该保持不变但运动方向反了。整体平滑移动了 (x, y) 之后，相邻点之间的位移方向与 theta 产生了矛盾——车辆的"朝向"和"运动方向"对不上，导致路径在物理上不可执行。

---

### 2.2 尝试：按换挡点切段平滑

**思路：** 既然整体平滑会破坏换挡点的 theta，那就先找出所有换挡点，把路径切成若干段，每段单独平滑，每段首尾固定不动。平滑后 theta 用相邻点的位移方向重新计算，而不是保留原值。

**实现细节：**
1. 遍历路径，找到 v 符号变化的位置（前进→倒车或倒车→前进），这些点就是换挡点
2. 按换挡点把路径切成若干段
3. 对每段用 `np.linspace` 加密插值（每 0.3m 一个点）
4. 每段独立做梯度下降平滑，首尾坐标固定
5. 平滑后根据位移方向重新计算 theta，并和原始 theta 比较判断前进/倒车

**结果：代码能跑通，但平滑效果几乎为零——"白瞎"。**

**根本原因（关键发现）：**

回头看 `hybrid_astar.py`：

```python
next_state = self.car.update_state(curr_state, v, phi, dt=1.0)
```

`dt=1.0`，`v=1.0`，所以 **Hybrid A\* 每一步走 1 米**。一次典型的泊车路径只有十几个状态点，相邻两点之间是 1m 的大跨步。

插值加密（`np.linspace`）在两个相邻点之间插了很多点，但这些插值点是**线性插值**（直线），相邻状态点之间本来就没有弯曲——直线段上根本没有折角可以平滑。平滑力的输入（相邻三点的夹角）接近 180°，力几乎为零，迭代收敛后路径几乎没有变化。

要让路径平滑有意义，需要把 `dt` 减小（比如 0.3m），让搜索路径本身就更密，相邻点之间有真实的折角。但这样搜索时间会大幅增加，在当前实验周期内不可行。

**结论：放弃路径平滑。`path_smoother.py` 文件保留但不集成到主流程。**

## 方向三：MPC 轨迹跟踪控制器（主线）

放弃路径平滑后，转向 MPC。MPC 本身就能输出平滑的连续控制量，让车辆沿参考路径行驶，比后处理平滑更正确。

### MPC 基本原理

每一步：
1. 获取当前车辆状态 $(x, y, \theta)$
2. 向前看 N 步（预测窗口），枚举控制序列 $(v_0, \varphi_0), ..., (v_{N-1}, \varphi_{N-1})$
3. 找一组控制序列，使预测轨迹最贴近参考路径，同时控制量变化不太剧烈
4. 只执行第一步控制 $(v_0, \varphi_0)$，移动后重新规划

与 Hybrid A\* 的区别：
- Hybrid A\*：$(v, \varphi)$ 是离散的 6 个固定值，用于搜索
- MPC：$(v, \varphi)$ 是连续实数，优化器可以找到任意中间值（如 `v=0.73, φ=0.21`），输出更平滑

**整体架构：**
```
Hybrid A*（离散搜索，路径带 v 符号）
    → interpolate_path（每 0.1m 插一个点）
    → MPC 滚动优化（scipy SLSQP）
    → 每步输出连续 (v, φ)
    → car.update_state 仿真执行
```

### 迭代过程详细记录

#### v1：车停在起点不动

**现象：** 跑完 300 步，MPC 轨迹（红线）就在起点，Final Pose 等于起始位置。

**排查：** 发现 `u0 = np.zeros(2 * self.N)`，初始猜测全为零。SLSQP 是局部优化器，从零出发时梯度也接近零（代价函数关于零控制是平坦的），优化器找不到下降方向，直接收敛在原地。

**修复：** 根据当前车辆朝向与参考点方向的夹角，判断应该前进还是倒车，将初始猜测设为 `v = ±0.8`：

```python
heading_diff = math.atan2(math.sin(ref0[2] - state[2]), math.cos(ref0[2] - state[2]))
v_init = 0.8 if abs(heading_diff) < math.pi / 2 else -0.8
u0 = np.array([v_init, 0.0] * self.N, dtype=float)
```

---

#### v2：前半段跟踪正常，换挡点后穿进障碍物

**现象：** 车从起点出发，前进段跟踪参考路径很好。但到达换挡点附近后，继续前进穿进了左侧邻车障碍物，没有执行倒车。

**排查：** v1 的初始猜测根据角度差判断前进/倒车，在换挡点附近两个方向角度差接近 90°，判断结果不稳定。MPC 一直用前进速度去跟踪倒车段的参考点，方向完全错误。

**修复：** 让 Hybrid A\* 在路径点里存入 v 符号（将路径点改为 4 元组 `(x, y, θ, v)`），MPC 直接读参考点的 v 符号来决定初始猜测方向：

```python
# hybrid_astar.py
next_with_v = (next_state[0], next_state[1], next_state[2], v)

# mpc_controller.py
v_sign = 1.0 if (len(ref0) < 4 or ref0[3] >= 0) else -1.0
u0 = np.array([v_sign * 0.8, 0.0] * self.N, dtype=float)
```

---

#### v3：ref_idx 卡在末尾，MPC 到不了终点

**现象：** MPC 跟踪到参考路径末尾附近后停住了，Final Pose 停在车位入口，没有倒进去。

**排查：** `_nearest_ref` 函数找到最近参考点后，当 ref_idx 到达路径末尾时，搜索窗口 `start_idx + 20` 没有更多点了，best_idx 一直停在最后几个点，MPC 的参考状态不再更新，优化陷入死循环。

**修复两处：**

1. `_nearest_ref` 加推进逻辑：如果当前点已经很近（< 0.5m），自动推进到下一个点

```python
if min_dist < 0.5 and best_idx < len(ref_path) - 1:
    best_idx = min(best_idx + 1, len(ref_path) - 1)
```

2. 到达路径末尾 10 个点内时，将所有参考点替换为路径终点，让 MPC 专注于到达终点：

```python
if idx >= len(ref_path) - 10:
    goal = ref_path[-1]
    ref_points = [goal] * self.N
```

#### v4：MPC 抄近路穿障碍物

**现象：** 加了碰撞惩罚之前，MPC 完全不跟参考路径，自己绕了一个大圈跑到地图边界外。加了碰撞惩罚后，仍然偶尔会走出一条与参考路径完全不同的路线穿过障碍物。

**根本原因：** Hybrid A\* 路径只有十几个稀疏点，相邻点间距约 1m。MPC 的预测窗口 N=5 步对应 2.5 秒，在这段时间内车辆可以走任意路线"到达"下一个参考点——哪怕穿墙也能在数学上更快到达参考点，优化器会选这条路。

**修复：** 在 `main_vertical.py` 中对路径做线性插值加密，将稀疏路径变成每 0.1m 一个点的密集路径：

```python
def interpolate_path(path, step=0.1):
    dense = [path[0]]
    for i in range(1, len(path)):
        p1, p2 = path[i-1], path[i]
        dx, dy = p2[0]-p1[0], p2[1]-p1[1]
        dist = math.sqrt(dx*dx + dy*dy)
        n = max(1, int(dist / step))
        for k in range(1, n):
            t = k / n
            dense.append((p1[0]+t*dx, p1[1]+t*dy, p1[2], p1[3]))
        dense.append(p2)
    return dense
```

路径加密后 MPC 的参考点更密集，每步只需要向前走 0.1m，大幅压缩了"抄近路"的空间。

---

#### v5：倒车段卡在车位入口进不去

**现象：** MPC 成功驾驶到车位入口附近，但倒车时停住了，Final Pose 停在入口正上方，不肯进入车位。

**原因分析：** 当时 `d_safe=1.2m`，`w_obs=20.0`。车位宽 2.5m，两侧各有 0.2m 间隙，实际可用宽度约 2.5m，左右各距障碍约 1.25m。而 `d_safe=1.2m` 意味着只要中心点距障碍 < 1.2m 就会受到惩罚。车位内部的距离场值恰好在 1.2m 附近，倒车进入车位的代价极高，MPC 宁愿停在入口也不肯进去。

**修复：** 降低 `d_safe` 和 `w_obs`：

```python
d_safe = 0.5   # 1.2 → 0.5
w_obs  = 10.0  # 20.0 → 10.0
```

这样车位内部距离场值（约 1.25m）远大于 d_safe，倒车进去不会受到惩罚。

---

#### v6：偏离中心线，车身角点碰障碍（碰撞盲区问题）

**现象（截图 3.png, 4.png）：** 部分案例倒车时偏离中心线，Final Pose 偏左碰左侧障碍，或偏右碰右侧障碍。

**根本原因（系统性缺陷）：**

碰撞惩罚只查车辆**中心点**到障碍的距离：

```python
dist = self._get_dist(px, py)   # 只查中心点
if dist < self.d_safe:
    J += self.w_obs * (self.d_safe - dist) ** 2
```

车宽 1.8m，`d_safe=0.5m`。当 MPC 倒车偏了 0.6m 时：
- 车辆中心点距障碍 ≈ 1.25 - 0.6 = 0.65m > 0.5m → **惩罚不触发**
- 车身右侧角点已经越过障碍边界 → **物理上已经碰撞**

这就是为什么 3.png 碰左、4.png 碰右——两个方向都会碰，是同一个盲区导致的系统性缺陷。

**尝试修复：** 将碰撞惩罚改为检查车身4个角点：

```python
corners = self.car.get_corners((px, py, ptheta))
for cx, cy in corners:
    dist = self._get_dist(cx, cy)
    if dist < self.d_safe:
        J += self.w_obs * (self.d_safe - dist) ** 2
```

---

#### v7：角点碰撞惩罚导致成功率下降，回退

**现象：** 改为角点碰撞惩罚后，8次测试中成功次数下降（约 4/8），比中心点版的 6/10 更差。

**原因：** 每次 SLSQP 优化中，代价函数需要计算 N=5 步 × 4个角点 = 20 次距离场查询，加上梯度估计，总计算量大幅增加。SLSQP 的 `maxiter=100` 内更难收敛到好的局部极值，直接卡死的概率上升。

**这是 SLSQP 框架内无法同时解决的矛盾：**
- 查中心点：成功率 60%，有碰撞盲区
- 查角点：无盲区，但成功率更低（约 50%）

**结论：回退到中心点碰撞惩罚。** 这个矛盾只能通过换更强的优化器（如 CasADi + IPOPT）或换控制策略（Pure Pursuit）来根本解决。

## 总结与反思

### 做成了什么
1. 实现了完整的 Hybrid A\* + MPC 泊车闭环系统
2. MPC 能够在约 60% 的随机起点下成功完成垂直泊车
3. 完整实现了 RS 路径族（48种），为后续工作打下基础
4. 系统性地尝试并排查了多种失败模式

### 没做成什么以及为什么
1. **搜索加速**：三种方法（numpy 向量化、RS 启发式、双向搜索）都未取得实质性突破。numpy 向量化反而更慢；RS 启发式提升不到 8%；双向搜索在相遇点产生朝向折角，SE(2) 空间里两棵树相遇需要位置+朝向同时兼容，实现复杂度过高而放弃
2. **路径平滑**：dt=1.0 导致路径点稀疏，平滑无处发力。根本解决需要先把 dt 降下来，但代价是搜索时间大幅增加
3. **MPC 无碰撞**：中心点惩罚有盲区，角点惩罚让成功率更低。SLSQP 这个优化器不适合这个场景

### 关键教训
- **RS 曲线不能替代搜索**：RS 是纯几何计算，不检碰撞，只能做启发式
- **numpy 向量化需要看数据规模**：小数据用 Python 循环更快
- **SLSQP 的根本局限**：局部优化器，初值敏感，在窄间隙场景容易卡死，增加碰撞约束后更难收敛
- **插值加密是关键**：路径稀疏时 MPC 会抄近路，必须先加密才能有效跟踪

### 下一步
1. **Pure Pursuit 控制器**：严格贴密集参考路径，天然继承 Hybrid A\* 的无碰撞性，成功率预计显著高于 MPC
2. **换优化器**：CasADi + IPOPT 支持真正的约束（而非惩罚项），能同时保证无碰撞和高成功率
3. **路径密化 + 重新尝试平滑**：把 dt 从 1.0 降到 0.3，路径点增密后平滑才有意义
