# Phase 4 实验记录：搜索加速、路径优化与 MPC 跟踪控制

> 这份文档记录 Phase 4 的完整实验过程：试了什么、现象是什么、为什么失败、最后为什么成功。重点不是只记结果，而是把思考链条保留下来，方便以后回看，也方便继续往下做。

---

## 1. 这一阶段到底要解决什么问题

到 Phase 3 为止，Hybrid A* 已经能在 SE(2) 空间里搜出**运动学可行**的泊车路径，但还存在两个明显问题：

1. **搜索速度慢**
   现在的启发式主要还是欧氏距离，在 SE(2) 这种大状态空间里不够强。
2. **只有规划，没有控制**
   路径是搜出来了，但还没有一个控制器真的去跟踪执行这条路径。

所以 Phase 4 的核心目标是三件事：

- 搜索加速
- 路径优化 / 平滑
- MPC 跟踪控制

这一阶段真正想回答的问题是：

> 能不能把“能搜出一条可行路径”升级成“车能稳定地把这条路径执行出来”？

## 3. 方向一：搜索加速

这一部分的目标很直接：

> Hybrid A* 已经能用，但太慢了，能不能加速？

### 3.1 尝试：把 `collision_checker.py` 改成 numpy 向量化

**动机**

碰撞检测是 Hybrid A* 里调用最频繁的函数之一，直觉上它很可能是性能热点。既然 Python 循环慢，自然会想到：能不能用 numpy 批量计算提速？

**做法**

- 用 `np.linspace` 批量生成车身边界上的采样点
- 用数组索引一次性查 `grid_map[ix_arr, iy_arr]`
- 替换掉原来逐点循环判断的写法

**现象**

结果比纯 Python 版本更慢。

**根本原因**

每次碰撞检测其实只检查大概 40 到 60 个点，这个数据量太小了。numpy 的优势要等到数据规模大时才能体现，而这里数组创建、类型转换、内存分配这些固定开销，反而比循环本身更贵。

**结论**

放弃，回退到纯 Python 版本。

**这一轮学到的东西**

numpy 不是自动加速器。是否更快，取决于数据规模。

---

### 3.2 尝试：直接用 Reeds-Shepp 路径代替 Hybrid A* 搜索

**动机**

RS 曲线本来就是支持前进和倒车的最短路径。如果它能直接从起点连到终点，那理论上可以跳过 Hybrid A*，速度会快很多。

**做法**

- 直接调用 `rs_path(start_pos, goal_pos, turning_radius)`
- 试图把 RS 路径当成完整泊车路径使用

**现象**

失败，路径经常直接穿过障碍物。

**根本原因**

RS 曲线只是**自由空间里的几何最短路径**。它考虑的是起点、终点、转弯半径，但完全不知道地图上哪里有障碍物。

在停车场景里，车位两边有邻车，RS 路径会直接穿过去。

**结论**

RS 曲线不能替代搜索，只能做启发式或者局部连接工具。

**这一轮学到的东西**

“满足运动学”不等于“能避障”。

---

### 3.3 尝试：把 Reeds-Shepp 距离作为启发式

**动机**

既然 RS 曲线不能直接拿来当路径，那它也许可以作为更强的启发式。因为它考虑了最小转弯半径，比欧氏距离更接近真实代价。

**做法**

- 实现完整的 RS 路径族
  - CSC：`LSL`、`RSR`、`LSR`、`RSL`
  - CCC：`RLR`、`LRL`
- 再用 `time_flip` 和 `reflect` 扩展变体
- 最后在所有候选中取最短长度作为启发式
- 为了不让每一步都算太慢，只在接近终点时切换到 RS 启发式

```python
h = rs_distance(next_state, self.goal, self.turning_radius) if euclidean < 5.0 else euclidean
```

**现象**

搜索方向确实更准了，尤其在接近终点的几米内更明显，但整体速度提升不到 8%。

**根本原因**

问题的主要瓶颈不是“启发式不够准”这么简单，而是整个 SE(2) 搜索空间本来就太大。启发式变好，只是让搜索方向更聪明，但展开节点数量并没有从根上大幅减少。

**结论**

这部分代码保留，因为方向是对的，但它不是决定性的加速手段。

**这一轮学到的东西**

更强的启发式有帮助，但未必能解决状态空间爆炸。

---

### 3.4 尝试：双向 Hybrid A* 搜索

**动机**

双向搜索是经典加速思路。二维图搜索里，从起点和终点同时扩展，理论上可以显著减少搜索深度。

**做法**

- 同时维护 forward 和 backward 两棵搜索树
- 每次扩展后检查两边是否在同一个离散桶里相遇
- 如果相遇，就把两段路径拼起来

**现象**

失败。拼接点出现明显折角，路径不可执行。

**根本原因**

在 SE(2) 里，相遇不只是位置对上，还要朝向也兼容。

- forward 树末端有一个朝向 `theta_f`
- backward 树末端有另一个朝向 `theta_b`

位置即便落在同一个格子里，这两个朝向往往也对不上。直接拼接，相当于要求车辆在某个点瞬间改头，违反运动学约束。

这就是双向 Hybrid A* 的本质难点：在 `(x, y, θ)` 空间里，“相遇”的条件比二维格子严得多。

**结论**

放弃，不保留。

**这一轮学到的东西**

二维搜索里好用的方法，搬到 SE(2) 里不一定成立。

---

### 3.5 搜索加速这条线的总结

| 尝试 | 结果 | 为什么没成为最终方案 |
|------|------|----------------------|
| numpy 向量化 | 更慢 | 数据规模太小，固定开销压过收益 |
| RS 直接代替搜索 | 失败 | 不感知障碍物 |
| RS 启发式 | 略有提升 | 提升太小，不是主瓶颈 |
| 双向 Hybrid A* | 失败 | SE(2) 下相遇点朝向不兼容 |

**总的结论**

这一阶段里，搜索加速没有拿到决定性突破。真正的主瓶颈仍然在 SE(2) 搜索本身的复杂度上。

---

## 4. 方向二：路径优化 / 平滑

Phase 3 的路径虽然可行驶，但本质上还是由离散运动原语拼起来的，存在明显折角和锯齿感。

这一部分想解决的是：

> 能不能把 Hybrid A* 的离散路径变成更平滑、更容易被控制器跟踪的轨迹？

### 4.1 尝试：整条路径一起做梯度下降平滑

**思路**

给整条路径上的所有点同时施加两种“力”：

- 平滑力：让相邻三点尽量趋向共线
- 障碍物排斥力：让路径远离障碍物边界

**做法**

- 用 `distance_transform_edt` 预计算距离场
- 用数值差分近似距离场梯度
- 迭代更新所有路径点的 `(x, y)`
- `theta` 保持原值不变

**现象**

失败。几何上路径动了，但朝向和位移方向对不上，整条路径变得不可执行。

**根本原因**

Hybrid A* 路径里有换挡点。换挡前后，车的朝向和运动方向之间不是简单一致关系。现在只改了 `(x, y)`，却保留了原始 `theta`，结果导致：

- 几何轨迹变了
- 姿态语义没变
- 位移方向和车身朝向发生冲突

也就是说，路径从“几何上更平滑”变成了“物理上不合法”。

**结论**

整体平滑不可用，放弃。

---

### 4.2 尝试：按换挡点切段，再分别平滑

**思路**

既然整条路径一起平滑会破坏换挡语义，那就按前进 / 倒车切段，每一段单独平滑。

**做法**

1. 先找出所有 `v` 符号变化的位置
2. 按换挡点把路径切成几段
3. 每段先插值加密到 `0.3m` 左右一个点
4. 每段独立平滑
5. 平滑后再根据位移方向重新计算 `theta`

**现象**

代码能跑，但几乎没有平滑效果。

**根本原因**

这是这一部分最关键的发现。回头看 `hybrid_astar.py`：

```python
next_state = self.car.update_state(curr_state, v, phi, dt=1.0)
```

`dt=1.0`，`v=1.0`，意味着 Hybrid A* 每一步大概走 1 米。整条泊车路径通常只有十几个状态点，点与点之间太稀疏。

后面虽然做了插值加密，但插进去的点只是直线插值，它们并没有真正增加路径的局部几何信息。原始路径本身就缺少可平滑的“细节”，所以平滑力几乎无处发挥。

说白了就是：

- 原路径太粗
- 插值只是补点，不是补曲率结构
- 没有足够的折角信息可以被平滑

**结论**

`path_smoother.py` 保留，但不接入主流程。

**这一轮学到的东西**

如果原始路径分辨率太低，后处理平滑很难真正救回来。

---

### 4.3 路径平滑这条线的总结

| 尝试 | 结果 | 根本问题 |
|------|------|----------|
| 整体平滑 | 失败 | 破坏了姿态 / 换挡语义 |
| 分段平滑 | 几乎无效 | 原始路径太稀疏，`dt=1.0` 太大 |

**总的结论**

这一部分真正暴露的问题，不是“梯度下降法不够好”，而是上游 Hybrid A* 输出的路径本身分辨率太低。

这个结论也直接推动了后面的转向：

> 与其指望离线平滑去补救，不如让控制器直接输出更平滑的连续控制量。

---

## 5. 方向三：MPC 跟踪控制器

路径平滑这条线效果不理想之后，主线自然转到了 MPC。

核心思路是：

- Hybrid A* 提供一条可行参考路径
- MPC 根据当前状态和未来参考点，滚动优化出连续控制量
- 最终目标不是“有路径”，而是“能把路径开出来”

整体流程：

```text
Hybrid A* 路径
-> 路径插值加密
-> MPC 滚动优化
-> 连续控制量 (v, phi)
-> 车辆仿真执行
```

这里和 Hybrid A* 的本质区别是：

- Hybrid A* 用的是离散的几个运动原语
- MPC 优化的是连续控制量

所以理论上 MPC 更适合做真实跟踪。

---

## 6. 基于 SLSQP 的 MPC：七轮迭代

这一部分是 Phase 4 最有价值的内容。因为真正的系统瓶颈，就是在这几轮里一点点暴露出来的。

### v1：车不动，停在起点

**现象**

跑完很多步后，MPC 轨迹几乎还在起点。

**根本原因**

初始猜测是全零：

```python
u0 = np.zeros(2 * self.N)
```

SLSQP 是局部优化器，从全零出发，很容易直接收敛到“不动”这个局部最优。

**修复**

根据当前朝向和参考点方向的夹角，先给一个前进或倒车的初始速度猜测：

```python
heading_diff = math.atan2(math.sin(ref0[2] - state[2]), math.cos(ref0[2] - state[2]))
v_init = 0.8 if abs(heading_diff) < math.pi / 2 else -0.8
u0 = np.array([v_init, 0.0] * self.N, dtype=float)
```

---

### v2：前半段正常，换挡后出问题

**现象**

前面跟踪得还可以，但到了换挡点附近，车没有正确倒车，而是继续前进，最后撞进障碍物。

**根本原因**

控制初始方向是根据角度差临时猜的，而换挡点附近这个判断很不稳定。结果就是：参考段其实应该倒车，但优化器还是以前进速度去追。

**修复**

让 Hybrid A* 路径直接带上 `v` 的符号，不再让 MPC 靠几何关系猜：

```python
next_with_v = (next_state[0], next_state[1], next_state[2], v)
```

MPC 直接读取这个符号：

```python
v_sign = 1.0 if (len(ref0) < 4 or ref0[3] >= 0) else -1.0
u0 = np.array([v_sign * 0.8, 0.0] * self.N, dtype=float)
```

---

### v3：到路径末尾附近停住，进不去终点

**现象**

车开到车位入口附近之后不再推进，停在外面。

**根本原因**

参考索引到了路径尾部后，后面没有新的参考点可以继续拉着 MPC 往前走。控制器反复盯着末尾附近几个点，等于“目标不再前进”。

**修复**

1. 如果离当前参考点足够近，就主动推进索引
2. 当接近路径末尾时，直接把未来参考点都替换成终点

```python
if min_dist < 0.5 and best_idx < len(ref_path) - 1:
    best_idx = min(best_idx + 1, len(ref_path) - 1)
```

```python
if idx >= len(ref_path) - 10:
    goal = ref_path[-1]
    ref_points = [goal] * self.N
```

---

### v4：MPC 抄近路，直接穿障碍物

**现象**

MPC 有时会走出一条和参考路径很不一样的路，甚至出现“抄近路穿墙”的现象。

**根本原因**

Hybrid A* 输出的原始路径点太稀疏，相邻点大约 1 米。MPC 预测窗口里，优化器会觉得“只要我能更快到后面的参考点就行”，于是会钻空子，选出一条在数学上更快、但并不是原本参考路径的路线。

**修复**

在送给 MPC 之前，先把路径线性插值加密到 `0.1m` 一个点：

```python
def interpolate_path(path, step=0.1):
    dense = [path[0]]
    for i in range(1, len(path)):
        p1, p2 = path[i-1], path[i]
        dx, dy = p2[0]-p1[0], p2[1]-p1[1]
        dist = math.sqrt(dx*dx + dy*dy)
        n = max(1, int(dist / step))
        for k in range(1, n):
            t = k / n
            dense.append((p1[0]+t*dx, p1[1]+t*dy, p1[2], p1[3]))
        dense.append(p2)
    return dense
```

这样参考路径更密，MPC 的自由度就被压住了。

---

### v5：车到了车位口，却不肯进去

**现象**

车能开到车位入口，但倒车时总是停在外面，不肯真正进车位。

**根本原因**

当时障碍物安全距离设得太大：

- `d_safe = 1.2`
- `w_obs = 20.0`

对优化器来说，进入车位的代价太高，停在入口反而更“划算”。

**修复**

把安全距离和障碍惩罚都降下来：

```python
d_safe = 0.5
w_obs  = 10.0
```

---

### v6：末段偏离中心线，碰左边或右边障碍

**现象**

一些案例里，车虽然基本倒进去了，但最终姿态偏左或偏右，导致角点碰到邻车。

**根本原因**

碰撞代价只看**车中心点**到障碍物的距离：

```python
dist = self._get_dist(px, py)
if dist < self.d_safe:
    J += self.w_obs * (self.d_safe - dist) ** 2
```

这就产生了一个系统性盲区：

- 车中心点看起来还安全
- 但车身角点实际上已经越界

也就是说，问题不是偶发，而是模型本身就没把车身宽度真正考虑进去。

**修复尝试**

改成检查 4 个角点：

```python
corners = self.car.get_corners((px, py, ptheta))
for cx, cy in corners:
    dist = self._get_dist(cx, cy)
    if dist < self.d_safe:
        J += self.w_obs * (self.d_safe - dist) ** 2
```

---

### v7：改成角点检测后，成功率反而下降

**现象**

本来以为角点检测能解决碰撞盲区，结果成功率不升反降。

**根本原因**

SLSQP 顶不住更复杂的代价函数。

- `N = 5`
- 每一步 4 个角点
- 一整个预测窗口要算很多次距离

再加上 SLSQP 本来就是数值梯度、局部优化，问题一复杂，就更容易卡死。

这一轮真正暴露出来的是一个结构性矛盾：

- 只查中心点：容易收敛，但不安全
- 查 4 角点：更安全，但更难优化

**结论**

在 SLSQP 这个框架里，很难同时要“高成功率”和“强碰撞感知”。

这其实就是整个 Phase 4 的真正转折点。

## 8. SLSQP 版本的最终结果

### 最终参数

```text
N=5, dt=0.5
w_pos=5.0, w_theta=2.0, w_ctrl=0.1, w_obs=10.0, d_safe=0.5
max_steps=500
终止条件：位置误差 < 0.3m，角度误差 < 5°
```

### 10 次测试结果

| 次数 | 起点朝向 | 结果 |
|------|---------|------|
| 1 | ~23° | 成功 |
| 2 | ~146° | 成功 |
| 3 | ~260° | 失败 |
| 4 | ~75° | 成功 |
| 5 | ~350° | 失败 |
| 6 | ~190° | 成功 |
| 7 | ~118° | 成功 |
| 8 | ~34° | 失败 |
| 9 | ~92° | 成功 |
| 10 | ~200° | 失败 |

**成功率：`6/10 = 60%`**

失败案例大多是起始朝向和目标差得比较大时，SLSQP 更容易掉进局部最优。

## 9. 到这一步，真正的瓶颈终于看清了

做到这里以后，有几个判断已经很明确了：

1. **主瓶颈已经不是“搜不出路径”**
   Hybrid A* 给出的参考路径已经基本够用。
2. **路径平滑不是当前最关键的突破口**
   因为原始路径太稀疏，后处理空间有限。
3. **真正卡住系统的是控制优化框架**
   SLSQP 在窄车位、强碰撞约束下太脆弱。

也就是说，下一步不该继续死磕 SLSQP 调参，而应该换框架。

---

## 10. 升级：CasADi + IPOPT + 4 角点碰撞检测

### 10.1 为什么必须升级

SLSQP 版本有两个根本问题：

1. **中心点碰撞盲区**
   车中心还安全，不代表车身不碰撞。
2. **对初值太敏感**
   一旦起点姿态不友好，就容易卡住。

这两个问题在 SLSQP 框架里基本是绑在一起的，很难同时解决。

所以最终思路不是继续小修小补，而是直接换优化器。

---

### 10.2 核心改动一：`SLSQP -> CasADi + IPOPT`

- CasADi 负责符号建模
- IPOPT 负责非线性优化
- 梯度不再靠数值差分，而是自动微分

而且整个 N 步优化问题在 `__init__` 里只构建一次，后面每一步只传参数进去求解：

```python
def _build_solver(self):
    U = ca.MX.sym('U', 2 * N)
    P = ca.MX.sym('P', 3 + N*3 + N*4)
    self._solver = ca.nlpsol('mpc', 'ipopt', nlp, opts)
```

这样速度反而更快，单步大概 `3 ms`。

---

### 10.3 核心改动二：碰撞检测从中心点升级到 4 角点

CasADi 里不太方便直接对距离场做标准符号查表，所以用了一个工程近似：

- 先用当前状态数值计算 4 个角点到障碍物的距离
- 再把这些值作为参数传给求解器
- 在短预测窗口里近似认为它们变化不大

```python
corner_d = self._corner_dists(state)
p = list(state[:3]) + ref_coords + corner_d * N
```

这不是最严格的物理建模，但对 `N = 5`、`dt = 0.5` 这样的短时域 MPC 来说，实际效果已经够好了。

### 10.4 参数对比

| 项目 | SLSQP 版 | CasADi + IPOPT 版 |
|------|----------|-------------------|
| 优化器 | `scipy SLSQP` | `CasADi + IPOPT` |
| 梯度 | 数值差分 | 自动微分 |
| 碰撞代价 | 中心点 | 4 角点 |
| `w_obs` | `10.0` | `15.0` |
| 单步耗时 | ~20ms | ~3ms |

## 12. Phase 4 真正建立起来的东西

这一阶段真正做成的，不只是“加了个 MPC”，而是得出了几个更深的结论：

### 12.1 关于搜索

搜索加速没那么容易。启发式变强有帮助，但并没有从根本上解决 SE(2) 状态空间的负担。

### 12.2 关于路径平滑

如果原始路径本身太稀疏，后处理平滑的空间非常有限。

### 12.3 关于控制

真正决定系统质变的，是控制优化框架升级：

- 从脆弱的局部优化，换到更强的非线性优化
- 从只看中心点，升级到真正感知车身包络

也就是说：

> Phase 4 最关键的突破，不是继续压榨旧搜索框架，而是把“路径执行”这件事真正做对了。

## 13. 这一阶段最重要的经验

1. **不要默认 numpy 一定更快**
   先看数据规模。
2. **不要把几何最短路径当成避障规划**
   RS 很重要，但它不是障碍环境里的完整规划器。
3. **不要指望后处理平滑修复一条太粗的路径**
   上游分辨率不够，后面很难救。
4. **优化器本身就是方法的一部分**
   同样叫 MPC，不同求解器、不同代价建模，效果可以差一个层级。
5. **窄车位场景必须考虑车身包络**
   只看中心点是不够的。
